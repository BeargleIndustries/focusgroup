# Contributing to focusgroup

Thanks for your interest in contributing!

## Getting Started

1. Fork the repository
2. Clone your fork locally
3. Install as a local Claude Code plugin:
   ```bash
   ln -s /path/to/your/fork ~/.claude/plugins/local/focusgroup
   ```
4. Make your changes
5. Test by running `/focusgroup` in Claude Code
6. Open a PR with example output showing the change

## What to Contribute

### Adding a New Review Protocol

Review protocols define how reviewers examine a specific target type. They live in `agents/protocols/`.

1. Create `agents/protocols/{name}-review-protocol.md`
2. Structure it as phases (numbered, sequential)
3. Each phase must specify:
   - Exact tool calls to make
   - Minimum evidence to collect
   - Expected output format
4. Add evidence rules at the top (prohibited vague phrases, citation requirements)
5. Add a fallback behavior table for unavailable tools
6. Add a dispatch entry in `commands/focusgroup.md` Phase 2.5

**Naming convention:** `{target-type}-review-protocol.md`

### Adding a New Methodology

Methodologies define multi-round review orchestration patterns. They live in `agents/methodologies/`.

1. Create `agents/methodologies/{name}-protocol.md`
2. This is an instruction document, NOT an agent — no frontmatter, no model tier
3. Include the header: "This document is an instruction set for the command orchestrator. It does NOT spawn agents."
4. Define round-by-round instructions:
   - Setup (directory creation)
   - Per-round reviewer spawn prompt templates
   - Inter-round aggregation logic
   - Final output generation format
5. Include a Synthesis Enhancement section
6. Add dispatch entry in `commands/focusgroup.md` Phase 2.5
7. Add the new flag to Phase 1 parsing and the mutual exclusivity check

**Naming convention:** `{methodology-name}-protocol.md`

**Key constraint:** Methodology documents provide instructions. The command is the sole agent spawner. No methodology document should contain agent spawning logic — only prompt templates for the command to use.

### Adding a New Output Format

Output generators create deliverables from synthesis results. They live in `agents/outputs/`.

1. Create `agents/outputs/{name}-generator.md`
2. Include agent frontmatter (name, description, model tier)
3. Define inputs (which files to read from `{out_dir}/`)
4. Define the output format with a template
5. Add a `--{flag}` in `commands/focusgroup.md` Phase 1 parsing
6. Add the generator spawn in Phase 7.5
7. Respect execution order: if your output depends on another (e.g., risk-matrix depends on action-plan), document the dependency

**Naming convention:** `{output-type}-generator.md`

**Model tier guidance:**
- Haiku: Simple transformation/formatting (SWOT, matrices)
- Sonnet: Complex analysis or generation (action plans, HTML reports)

### Adding Preset Persona Files

Example persona YAML files live in `examples/`.

1. Create `examples/personas-{type}.yaml`
2. Include exactly 5 personas per file
3. Each persona needs: name, slug, title, archetype, expertise (array), perspective, review_priorities (3 items), tone, rating_bias
4. Personas should feel like real professionals, not caricatures
5. Include diversity within the persona type (e.g., a security panel has both offensive and defensive roles)

**Naming convention:** `personas-{type}.yaml`

### Agent Improvements

Agent prompts in `agents/` can always be refined. If you find patterns that produce more specific, evidence-backed reviews:

- Update the relevant agent file
- Test with at least 2 target types
- Include before/after example output in your PR

### Bug Reports

Open an issue with:
- The `/focusgroup` command you ran (with flags)
- What you expected
- What happened instead
- Any error output
- The `meta.json` from the output directory (if available)

## File Naming Conventions

| Directory | Pattern | Example |
|-----------|---------|---------|
| `agents/` | `{role}.md` | `focus-reviewer.md` |
| `agents/probes/` | `{target}-probe.md` | `website-probe.md` |
| `agents/protocols/` | `{type}-review-protocol.md` | `website-review-protocol.md` |
| `agents/methodologies/` | `{method}-protocol.md` | `delphi-protocol.md` |

| `report.json` | Command | `--format json` flag |

## Architecture Notes

- **Command is sole orchestrator:** `commands/focusgroup.md` spawns all agents. No agent spawns sub-agents.
- **Runtime document loading:** Protocols and methodologies are loaded at runtime via Read tool. They are NOT inlined into the command.
- **Command line budget:** `commands/focusgroup.md` must stay under 550 lines.
- **Agent model tiers:** haiku (probes, simple generators), sonnet (reviewers, complex generators), opus (director only).

## Guidelines

- Keep prompts clear and specific — vague instructions produce vague reviews
- Test with at least 2 different target types
- Persona definitions should feel realistic
- Maintain structured output formats — downstream tools may parse them
- New features should be additive — don't break existing flag behavior

## Pull Requests

1. Open an issue first to discuss the change
2. Keep PRs focused — one feature or fix per PR
3. Update README.md and SKILL.md if you add new features
4. Include example output if you change templates or prompts
5. Verify the command stays under 550 lines
